## Project Overview

`flutter_twilio` is a Flutter plugin that exposes Twilio Programmable Voice
(VoIP) and Twilio's REST SMS API through a single `FlutterTwilio.instance`
facade. Supported platforms: **Android + iOS only** (Web and macOS are not
supported).

The native bridge is generated by [Pigeon](https://pub.dev/packages/pigeon)
— schema in `pigeons/voice_api.dart`, generated output checked in under
`lib/src/voice/generated/`, `ios/Classes/Generated/`, and
`android/src/main/kotlin/com/dev/flutter_twilio/generated/`.

Current version: `0.2.0`. CI pins Flutter `3.41.0` (stable).

## Architecture

### Dart layer

All public API lives under `lib/` and is re-exported from
`lib/flutter_twilio.dart`:

- `FlutterTwilio` (`lib/src/flutter_twilio.dart`) — the singleton facade.
  `init(accountSid:, authToken:, twilioNumber:)` stashes Twilio credentials
  in-memory and lazily wires up `.voice` + `.sms`.
- `VoiceApi` (`lib/src/voice/voice_api.dart`) — abstract voice surface.
  Implemented by `VoiceImpl` (`lib/src/voice/voice_impl.dart`), which
  adapts the generated Pigeon `VoiceHostApi` and translates
  `PlatformException`s into typed `VoiceException` subtypes. Implements
  `VoiceFlutterApi` to receive native → Dart event callbacks and broadcasts
  them on the `events` stream as `Call` snapshots.
- `SmsApi` (`lib/src/sms/sms_api.dart`) — abstract SMS surface. Implemented
  by `SmsClient` (`lib/src/sms/sms_client.dart`) — a pure-Dart Twilio REST
  client built on `package:http`. No native bridge involved.
- Models: `Call`, `ActiveCall`, `CallEvent` (`lib/src/voice/models/`),
  `Message` (`lib/src/sms/models/`).
- Errors: `VoiceException` + 5 concrete subtypes
  (`lib/src/voice/errors.dart`); `TwilioSmsException`
  (`lib/src/sms/errors.dart`). Both implement a shared
  `FlutterTwilioException` interface.

### Native bridge: Pigeon

The schema source of truth is `pigeons/voice_api.dart`. It declares:

- Enums `CallDirection`, `CallEventType`.
- Data classes `ActiveCallDto`, `CallErrorDto`, `CallEventDto`,
  `PlaceCallRequest`.
- `@HostApi() VoiceHostApi` — every method the Dart side calls on the
  native implementation (setAccessToken, register, place, answer, hangUp,
  setMuted/OnHold/Speaker, sendDigits, getActiveCall, hasMicPermission,
  requestMicPermission, etc).
- `@FlutterApi() VoiceFlutterApi` — the `onCallEvent` callback natives
  invoke to push events back to Dart.

Regenerate after any schema change:

```bash
./tool/pigeon_generate.sh
```

Generated outputs (all checked in, kept in sync with the schema):

| Generated file | Consumer |
|---|---|
| `lib/src/voice/generated/voice_api.g.dart` | `VoiceImpl` |
| `ios/Classes/Generated/VoiceApi.g.swift` | `FlutterTwilioPlugin` + `TVCallHandler` |
| `android/src/main/kotlin/com/dev/flutter_twilio/generated/VoiceApi.g.kt` | `FlutterTwilioPlugin` + handlers |

### Error model

`CallErrorDto { code, message, details }` is the wire shape. Both native
sides have a `FlutterTwilioError` helper
(`android/.../FlutterTwilioError.kt`, `ios/Classes/FlutterTwilioError.swift`)
that converts internal failures into the canonical `code` strings the Dart
layer matches on in `VoiceException.fromPigeon`. The canonical codes are
listed in `lib/src/voice/errors.dart`:
`not_initialized`, `missing_permission`, `invalid_argument`, `invalid_token`,
`no_active_call`, `call_already_active`, `twilio_sdk_error`,
`audio_session_error`, `registration_error`, `connection_error`,
`bluetooth_unavailable`, `wired_unavailable`, `audio_route_failed`,
`tone_asset_not_found`, `notification_permission_denied`.

When adding a new error case: add the `code` string here first, then add a
subtype in `lib/src/voice/errors.dart`, extend the switch in
`VoiceException.fromPigeon`, and use the matching
`FlutterTwilioError.<factory>` on both native sides.

### Native platforms

| Platform | Entry point | Key SDK | Notes |
|---|---|---|---|
| Android | `android/src/main/kotlin/com/dev/flutter_twilio/FlutterTwilioPlugin.kt` | Twilio Voice SDK v6.9.0 | Custom in-app calling + FCM, min SDK 26 |
| iOS | `ios/Classes/FlutterTwilioPlugin.swift` | Twilio Voice SDK v6.13.0 | CallKit + PushKit VoIP notifications, min iOS 13 |

**Android** (`android/src/main/kotlin/com/dev/flutter_twilio/`):
`FlutterTwilioPlugin.kt` registers the `VoiceHostApi` implementation and
fans calls out to per-domain handlers under `handler/`:

- `TVCallMethodHandler` — place/answer/hangUp + call controls
- `TVAudioMethodHandler` — speaker/mute/hold
- `TVRegistrationMethodHandler` — access-token registration
- `TVPermissionMethodHandler` — mic permission queries

Call state and audio routing live in `service/`:

- `TVCallManager` — orchestrates Twilio SDK `Call` / `CallInvite`
- `TVCallState` — single source of truth for the current call
- `TVCallAudioService` — foreground service keeping audio routing stable
- `TVAudioManager` — speakerphone + mute helpers

Push: `fcm/VoiceFirebaseMessagingService.kt` handles incoming Twilio VoIP
pushes. Events flow out via `TVEventEmitter` → `VoiceFlutterApi.onCallEvent`.

Calls run **in-app** — the previous `ConnectionService` / `TelecomManager`
integration is removed. The system dialer does not appear, and
`MANAGE_OWN_CALLS` / phone-account setup are no longer required. See
`MIGRATION.md` for details.

**iOS** (`ios/Classes/`): `FlutterTwilioPlugin.swift` is the main
`FlutterPlugin` / `PKPushRegistryDelegate` / `CXProviderDelegate`. It
installs a `VoiceHostApi` implementation (delegating to `TV*Handler`
classes) and keeps a `VoiceFlutterApi` reference for emitting events via
`TVEventEmitter`. Concern-specific helpers:

- `TVAudioHandler` — AVAudioSession + speaker
- `TVCallHandler` — call lifecycle, implements most `VoiceHostApi` calls
- `TVCallDelegate` — Twilio SDK `CallDelegate`
- `TVCallKitDelegate` — CallKit `CXProviderDelegate` glue
- `TVPushDelegate` — PushKit VoIP push handling
- `TVNotificationDelegate` — user-facing notifications (missed calls etc.)
- `TVPermissionHandler` — mic permission
- `TVRegistrationHandler` — access-token registration

### Event streaming

Native → Dart events flow through the generated `VoiceFlutterApi`. On
Android, `TVEventEmitter.kt` holds the `VoiceFlutterApi` reference and
serializes emits onto the Flutter main handler; on iOS,
`TVEventEmitter.swift` does the same with `DispatchQueue.main`. On the
Dart side, `VoiceImpl` implements `VoiceFlutterApi.onCallEvent` and
broadcasts every event on the `events` stream as a
`Call { event: CallEvent, active: ActiveCall? }`; any
`CallEventType.error` is converted into a `VoiceException.fromPigeon` and
surfaced as `onError` on the stream (not as a `Call` event).

## Platform Registration

`pubspec.yaml` declares:

- Android: `com.dev.flutter_twilio.FlutterTwilioPlugin`
- iOS: `FlutterTwilioPlugin`

There are no `web:` or `macos:` entries — those platforms are intentionally
not supported.

## Localization

Android string resources live in `android/src/main/res/values/` and
`values-es/`. When adding new user-facing strings on Android, add them to
both `strings.xml` files (English + Spanish Latin American).

For iOS localization, check `*.lproj` directories under `example/ios/Runner/`
(e.g. `en.lproj/Localizable.strings`).

## Key Constraints

- **No Web, no macOS.** The plugin is Android + iOS only; the old
  `twilio-voice.js` / WKWebView bridge is gone.
- **Android min SDK**: 26 (Android 8.0).
- **iOS deployment target**: 13.0.
- **Pigeon generated files are checked in** — regenerate with
  `./tool/pigeon_generate.sh` after editing `pigeons/voice_api.dart`, and
  commit the diff alongside the schema change. CI enforces that the
  generated files stay in sync (see `.github/workflows/flutter-build-test.yml`).
- **No ConnectionService** on Android — calls run in-app. See
  `MIGRATION.md` before touching Android call routing or permissions.
- **Credentials are in-memory only.** `FlutterTwilio.init(accountSid,
  authToken, twilioNumber)` is required before any voice/sms call; the
  plugin never persists them.
